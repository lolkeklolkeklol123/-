
Цель лабораторной работы — изучить плохие и хорошие практики написания CI/CD пайплайнов, а также понять, как исправления влияют на стабильность, скорость и безопасность сборки.

В работе представлены:
- Плохой CI/CD (`bad-ci.yml`) с намеренно плохими практиками
- Хороший CI/CD (`good-ci.yml`) с исправленными проблемами
- Отчёт с анализом каждой практики

---

## Плохие практики в `bad-ci.yml`

### 1. Использование `ubuntu-latest` и `actions/checkout@master`
**Почему плохо:**
- Эти версии меняются со временем, что может привести к неожиданным сбоям
- Нестабильная среда делает сборку непредсказуемой

**Исправление в `good-ci.yml`:**
- Используются фиксированные версии `ubuntu-22.04` и `actions/checkout@v4`

**Результат:**
- Пайплайн стал детерминированным и стабильным

---

### 2. Установка зависимостей через `apt` с sudo
**Почему плохо:**
- Медленно
- Используются права суперпользователя без необходимости
- Каждая сборка заново устанавливает зависимости

**Исправление:**
- Использован `actions/setup-python@v5` для корректной установки Python и pip

**Результат:**
- Быстрее, безопаснее, чище

---

### 3. Отсутствие кеширования зависимостей
**Почему плохо:**
- Каждый запуск скачивает все зависимости заново
- Занимает лишнее время и ресурсы

**Исправление:**
- Добавлен `actions/cache` для pip

**Результат:**
- Сборка ускорилась
- Использование ресурсов уменьшилось

---

### 4. Ручная работа с артефактами
**Почему плохо:**
- Логи и артефакты не сохраняются системой
- Непросто получить их через интерфейс GitHub

**Исправление:**
- Использован `actions/upload-artifact@v4`

**Результат:**
- Логи автоматически сохраняются и доступны для скачивания через GitHub Actions

---

## Вывод

Исправление плохих практик привело к:
- Повышению стабильности пайплайна
- Ускорению сборки
- Безопасной установке зависимостей
- Удобному доступу к артефактам

Таким образом, хорошо написанный CI/CD пайплайн становится надежным и поддерживаемым.
