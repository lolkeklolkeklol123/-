# Отчет по лабораторной работе №2: Сравнение сервисов Amazon Web Services и Microsoft Azure. Создание единой кросс-провайдерной сервисной модели.

## 1. Цель работы

Научиться анализировать облачные сервисы, не привязываясь к конкретному вендору (например, только к AWS или Azure). Сформировать общее понимание, что такое публичное облако и какие сервисы в нём есть.

## 2. Исходные данные

Для работы использовался файл с данными о расходах Microsoft Azure. В нём были такие поля:

Meter Category и Meter Sub-Category — общая категория и подкатегория услуги.
Meter Name — название конкретной услуги или метрики, за которую идёт плата.
Consumed Service — какой сервис Azure использовался (например, Microsoft.DBforPostgreSQL).

## 3. Ход работы

Мы взяли ту же самую модель, которую сделали для AWS в первой лабораторной (5 уровней: IT Tower → Service Family → Service Type → Service Sub Type → Service Usage Type) и применили её к данным Azure.

Главная задача — определить, какую функцию выполняет сервис Azure, и отнести его к уже готовой категории из модели AWS.
Например:

В биллинге Azure мы нашли и разобрали такие группы сервисов:

**Управляемые базы данных** (Azure Database for PostgreSQL, MySQL)
Готовые облачные базы, за которыми не нужно следить: масштабирование, обновления и бэкапы делает Azure.
Аналоги в РФ: Yandex Managed Service for PostgreSQL/MySQL, Cloud Databases в VK Cloud.

**Платформы для анализа данных** (Azure Databricks, HDInsight)
Инструменты для работы с большими данными и машинным обучением на основе Apache Spark, Hadoop и других технологий.
Аналоги: Yandex Data Processing, VK Cloud Big Data.

**Интеграция данных** (Azure Data Factory)
Сервис для настройки автоматических процессов переноса и преобразования данных между разными системами.
Аналоги: Прямого аналога в РФ нет, но похожие задачи решают Yandex DataSphere или VK Cloud Data Transfer.

**Потоковая аналитика** (Azure Stream Analytics)
Обработка данных в реальном времени (например, с датчиков или сайтов) с помощью SQL-подобных запросов.
Аналоги: Yandex Data Streams.

**Мониторинг и оповещения** (Azure Monitor)
Сбор метрик и логов, настройка алертов (уведомлений) при проблемах с приложениями или инфраструктурой.
Аналоги: Yandex Monitoring, Cloud Monitoring в VK Cloud.

**Резервное копирование** (Azure Backup)
Автоматическое создание бэкапов виртуальных машин, баз данных и файлов с централизованным управлением.
Аналоги: VK Cloud Backup.

**Кэш в памяти** (Azure Cache for Redis)
Быстрый кэш для ускорения работы приложений и снижения нагрузки на базы данных.
Аналоги: Yandex Managed Service for Redis, Cloud Databases в VK Cloud.

**Вычисления** (Виртуальные машины Azure, Azure Batch)
Виртуальные серверы и служба для запуска крупных параллельных задач (например, рендеринг или моделирование).
Аналоги: Yandex Compute Cloud, VK Cloud Compute.

**Хранилища для аналитики** (Azure Data Lake Store)
Место для хранения огромных объёмов структурированных и неструктурированных данных под аналитику и ML.
Аналоги: Yandex Object Storage (с соответствующими настройками).

**Сети доставки контента** (Azure CDN)
Ускорение загрузки статики (картинки, видео, скрипты) за счёт кэширования на серверах ближе к пользователям.
Аналоги: Yandex Cloud CDN.

В итоге получилась единая таблица, где сервисы AWS и Azure разложены по одним и тем же категориям. Это позволяет сравнивать, сколько тратится, например, на базы данных или аналитику в разных облаках.


## 4. Вывод

Мы создали единую модель, которая работает и для AWS, и для Azure. Главный результат — теперь можно сравнивать сервисы разных облаков по одним и тем же категориям, что помогает анализировать затраты и выбирать подходящие решения независимо от вендора.
